# Bridge Chrome Extension Architecture Guide

This guide is the source-of-truth implementation map for Bridge's Chrome extension architecture. It complements `CONTEXT.md`, which remains a glossary only.

## Architecture Rule

Organize code by runtime boundary first, then by feature. Chrome extension code does not all run in one JavaScript environment:

- The MV3 service worker is the event-driven extension coordinator and has no page DOM.
- The side panel is an extension page for user-visible controls and dashboard state.
- Content/page-runtime code works against the original page DOM and must preserve Guide-Only Assistance.
- Backend code runs in Node and keeps release-oriented provider credentials out of the extension.
- Domain code should not import Chrome APIs, DOM globals, or backend-only modules.

## Folder Responsibilities

### `src/extension/service-worker/`

Owns MV3 extension event orchestration. The service worker registers Chrome listeners, routes messages, coordinates Guidance Session lifecycle, asks providers for plans, injects page code, and broadcasts Session Dashboard changes.

Rules:

- Keep `main.ts` as the composition root.
- Do not put page-DOM traversal or overlay DOM construction here long term.
- Persist Guidance Session state through storage, not long-lived globals.
- Treat `chrome.scripting.executeScript` as a boundary call into page-runtime code.

Recommended next modules:

- `message-router.ts`
- `session-lifecycle.ts`
- `session-store.ts`
- `tab-events.ts`
- `provider-client.ts`
- `page-injection.ts`

### `src/extension/sidepanel/`

Owns the user-visible extension page. It renders the Session Dashboard, provider settings, Task Request form, Task Clarification answer flow, auto-refresh toggle, and Session End action.

Rules:

- Side panel code should communicate with the service worker through typed message constants.
- Do not duplicate Guidance Session lifecycle decisions here.
- Do not store provider credentials beyond the existing demo/local storage boundary.

### `src/extension/content/`

Reserved for code that runs against the original browser page. This includes Page Snapshot collection, Planning Payload reduction, overlay rendering, Page Target matching, Target Recovery, and Page State Change observation.

Rules:

- Content code may inspect, highlight, scroll, explain, and observe.
- Content code must not click, type, submit, purchase, delete, confirm, or otherwise act for the user.
- Do not send form values, full raw DOM, or screenshots to model providers by default.

Recommended modules:

- `page-snapshot.ts`
- `planning-payload.ts`
- `overlay.ts`
- `target-matching.ts`
- `page-state-observer.ts`

### `src/extension/platform/`

Reserved for wrappers around `chrome.*` APIs.

Rules:

- Domain and provider modules should not call Chrome globals directly.
- Wrappers should be thin and testable.
- Prefer wrapper-level tests for storage, message, tab, and scripting behavior.

Recommended modules:

- `runtime.ts`
- `tabs.ts`
- `storage.ts`
- `scripting.ts`
- `side-panel.ts`

### `src/domain/`

Owns Chrome-independent Guided Task Mode behavior and types.

Good fits:

- Plan Contract validation
- Guidance Plan modes and step caps
- Planning Payload privacy checks
- Completed Step History rules
- Plan Refresh rules that do not need Chrome APIs

Rules:

- No `chrome.*`
- No DOM APIs
- No network calls
- No backend environment variables

### `src/providers/`

Reserved for plan-creation provider adapters.

Rules:

- Providers return Plan Contract JSON.
- Providers do not own Guidance Session lifecycle.
- Providers do not render overlays or inspect pages.
- Provider-specific request shape, model defaults, credentials, and error text stay behind this boundary.

Recommended modules:

- `provider.ts`
- `backend-provider.ts`
- `gemini-provider.ts`
- `openai-provider.ts`

### `src/shared/`

Owns tiny cross-runtime constants and types, such as message names and storage keys.

Rules:

- Keep this folder small.
- Do not put behavior here just because two modules need it; use `src/domain/` for domain behavior.

### `src/backend/`

Owns the local backend proxy. It uses Node's built-in HTTP server and keeps Codex credentials out of the MV3 extension.

Rules:

- Keep `/health` and `/guidance-plan` contracts stable.
- Do not introduce Express or a backend framework as part of ordinary extension refactors.
- Do not log Planning Payload bodies, raw model responses, form metadata details, or sensitive URL query strings.
- Validate that Planning Payloads do not include user-entered form values before calling providers.

## Dependency Direction

Allowed dependency direction:

```text
extension/service-worker -> extension/platform
extension/service-worker -> domain
extension/service-worker -> providers
extension/service-worker -> shared
extension/sidepanel       -> shared
extension/content         -> domain
providers                 -> domain
providers                 -> shared
backend                   -> domain
backend                   -> shared
```

Avoid these dependencies:

- `domain -> extension`
- `domain -> backend`
- `domain -> providers`
- `providers -> extension/sidepanel`
- `extension/content -> backend`
- `extension/sidepanel -> content`

## Build And Runtime Output

Run:

```bash
npm run build
```

The build emits:

```text
dist/
  background.js
  sidepanel.html
  sidepanel.css
  sidepanel.js
  backend/
    server.cjs
  domain/
    guidance-contract.mjs
```

`manifest.json` loads `dist/background.js` and `dist/sidepanel.html`. Chrome should still load the repository root as the unpacked extension folder.

## Validation Expectations

Before handing off extension architecture changes:

```bash
npm run typecheck
npm test
npm run build
node --check dist/background.js
node --check dist/sidepanel.js
node --check dist/backend/server.cjs
```

Manual Chrome validation should still use normal `http://` or `https://` pages. Chrome blocks injection into internal pages such as `chrome://extensions`, the Chrome Web Store, and some browser-owned pages.

## Refactor Sequence For `service-worker/main.ts`

`src/extension/service-worker/main.ts` is intentionally still the current orchestration source. Shrink it in small vertical slices:

1. Move message dispatch into `message-router.ts`.
2. Move storage access into `session-store.ts`.
3. Move tab activation, navigation, and removal handlers into `tab-events.ts`.
4. Move provider selection and model calls into `src/providers/`.
5. Move page snapshot and overlay functions into `src/extension/content/`.
6. Move `chrome.scripting.executeScript` calls into `page-injection.ts`.
7. Remove `// @ts-nocheck` only after each extracted boundary has strict types and behavior tests.

At every step, preserve Guide-Only Assistance: Bridge may guide and observe, but the user performs every page action.
